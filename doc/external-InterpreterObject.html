External: InterpreterObject

InterpreterObject

An interpreter object is a user created object that has one or more external:InterpreterObject#interpreterMethods created using an InterpreterMethodFactory. This is not a description of a real class, neither is it an interface that can be implemented. Rather, the methods of this class are descriptions of the type of interpreter methods that an interpreter method factory can create for an interpreter object. Normaly, all interpretation callback functions supplied when creating interpreter methods will run as methods of their interpreter object, i. e. this will be bound to the interpreter object inside the interpretation body. However, external:InterpreterObject#methodFactoryTypeInterpreterMethod may be used to set the this binding to another object. See also external:ThisBinding.

atomTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

An atom type interpreter method is a method of an external:InterpreterObject that has been made by passing a regular expression and possibly an external:ThisBinding#atomInterpretation to InterpreterMethodFactory#atom. It is a type of external:InterpreterObject#interpreterMethod that parses and interprets its input text with its regular expression and its interpretation. Parsing is done by trying to match the regular expression at the position in the text where the last interpreter method finished to parse. Matches starting at other positions will not be concidered. The full match (corresponding to match[0]) will be parsed, so capturing groups will not have any effect. It returns the parsed text or, if it has an interpretation, the result of calling that interpretation as a method of the interpreter object with the parsed string. Inside the body of the interpretation, this will be bound the the object of its interpreter method, as if it was called as a method of the same object. If the regular expression can't be matched at the current position in the text, the method fails to parse and the interpretation isn't run.

emptyTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

An empty type interpreter method doesn't parse anything. It just runs the callback function interpretation passed to its factory function, InterpreterMethodFactory#empty, as if it was a method of the same object, and returns the result. In general, interpreterObject.emptyTypeInterpreterMethod("") should be equivalent to interpretation.call(interpreterObject).

groupTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

A group type interpreter method is an external:InterpreterObject#interpreterMethod that parses a group of other interpreter methods and regular expressions defined in its factory, InterpreterMethodFactory#group, one at the time. If one of the regular expressions or interpreter methods fails to parse, then the whole method fails to parse and the text pointer is restored, see external:InterpreterObject#interpreterMethod. It is also important to note that none of the interpretations of the parts is run before all parts has been successfully parsed, so it is never necessary to "clean up" after a half parsed group. If the group was defined with an external:ThisBinding#groupInterpretation it will call that with the results of its parts as arguments, and return the result. Otherwise it will return an InterpreterMethodFactory#MultiPropertyObject-like object with properties named as the interpreterMethodNames containing the results of the parts of the method.

InterpreterMethodResult

(abstract) interpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

This is a general description that applies to all interpreter methods. An interpreter method interprets text. When it is called from the outside code, it interprets the text of its first argument. When it is called as a part of another interpreter method, it parses as much of the text as it can, starting from the point in the text where the last interpreter method finished parsing.

If an interpreter method fails to parse, it will return the text pointer so that the next method will start parsing at the same position as the current one started before it failed. All interpreter methods are constructed never to be "half parsed"; if one of its parts fail, then the text pointer will be restored to where the interpreter method started parsing, even if some of the parts before the failing part were successfully parsed. Also, no interpretations of the descendants of an interpreter method will be run until all parts have been successfully parsed. Acctually, parsing of all the interpreter methods and running the interpretations of the successfull parsings is done in two completely separate steps. That way it is ensured that nothing that shouldn't be run is run.

Most interpreter methods comes in two different types. The first type is constructed without supplying a description of how the interpreter method should interpret the result of its parts. Then the method will return the results of its parts in some kind of container, like an array (for quantifiers), an object with properties with the same names as the interpreter methods (for groups), a string (for atoms) or just the result of the part (for opt, wrap and select). Interpreter methods of this type will create a parse tree with arrays and objects and strings (from the atoms) as its leafs.

The other type of interpreter method is when it is constructed with an interpretation. Then the interpreter method acts like a proxy for its interpretation. Instead of returning the results of its parts, the interpreter method will call its interpretation with those results, and then return the result of that call. An interpretation is meant to be thought of as a regular method of the same object as its interpreter method, but with enhanced abilities. Instead of just beeing called with arguments, they can create their own arguments by calling the methods named when they were constructed, or, in the case of atoms, getting their arguments by parsing text with an associated regular expression. Most importantly, interpretations will be run with this bound to the same object as their interpreter method. This way, interpretations function exactly like normal methods, but with the extra ability to, before they are called, compute their own arguments that they should be called with, by calling their parts.

longestTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

A longest type interpreter method is a type of external:InterpreterObject#interpreterMethod on a user created external:InterpreterObject constructed by providing the interpreterMethodNames of its parts to InterpreterMethodFactory#longest.

It is similar to an external:InterpreterObject#orTypeInterpreterMethod with the difference that it tries to parse all of its parts from the current position in the text, unlike or type interpreter methods that doesn't try to parse any more alternatives once it has found one that succeeds. It only runs the one of its parts that has succeeding parsing the most text so none of the other parts descendant interpretations will be run, even if they have parsed successfully. If two or more parts parses the same amount of text, then the first of them will be shoosen, to mimic the behaviour of or. The result of that part will also be the result of the whole interpreter method.

methodFactoryTypeInterpreterMethod(text, printDebuggingMessagesopt) → {function}

This external:InterpreterObject#interpreterMethod returns a function doing the same thing as the part named in its factory, InterpreterMethodFactory#methodFactory, would have done, but with the possibility to use another value for this in the interpretations of its descendant parts. The new value of this is supplied by running the instruction as a method of the external:ThisBinding object, by using call or apply, or by assigning it as a (temporary) property of the this binding before calling it. If the value of this should not be changed then the interpreter object has to be supplied in one of the beforementioned ways. If the returned function is called directly, interpretations of any descendant parts will get undefined (or worse, window) as the value for this. The returned function takes no arguments (at the moment, this might change).

optTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpretationMethodResult}

An opt type interpreter method is a type of external:InterpreterObject#interpreterMethod meant to be on an external:InterpreterObject created by the user. It is the result of calling InterpreterMethodFactory#opt with the interpreterMethodName of its only part and an external:ThisBinding#optFallbackReplacement.

Among the different types of interpreter methods, this is the equivalent of the ? quantifier in regular expressions, with the addition of having a defined behaviour if there is no part. The result of an interpreter method of this type is either the result of the part, if it could be parsed, or the result of its fallback replacement, if it couldn't be parsed, or undefined, if it couldn't be parsed and it has no fallback replacement. Inside the body of the interpretation, this will be bound to the object of the interpreter method.

orTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

An or type interpreter method is a type of external:InterpreterObject#interpreterMethod on a user created external:InterpreterObject constructed by providing the interpreterMethodNames of its parts to InterpreterMethodFactory#or. The behaviour of an or type interpreter method resembles the || operator. It tries to parse its parts, one at a time, until one of them parses successfully, and returns the result of that part. Just like || it doesn't try to parse any more parts if one has succeeded, and only the interpretations of the succeeding part will be run. If all of its parts fails to parse, then the whole interpreter method fails to parse.

plusTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpretationMethodResult}

A plus type interpreter method is a type of external:InterpreterObject#interpreterMethod meant to be put on a external:InterpreterObject created by the user. It is the result of calling InterpreterMethodFactory#plus with the interpreterMethodName of its only part and optionally a regular expression that will act as a delimiter.

Among the different types of interpreter methods, this is the equivalent of the + quantifier in regular expressions. It lets its part parse the text over and over until it can't parse anymore, optionally skipping over a delimiter regular expression in between the parsings. Just like the regex +, it needs to parse its part at least once to be successfully parsed.

The result of an interpreter method of this type is an array with the results of the repeated parsings of its part, if it wasn't defined with a external:ThisBinding#plusInterpretation. If it was, the result will be the result of calling its interpretation as if it was a method of the same object with the abovementioned array as the only argument. Inside the body of the interpretation, this will be bound to the object of the interpreter method.

wrapTypeInterpreterMethod(text, printDebuggingMessagesopt) → {InterpreterMethodResult}

A wrap type interpreter method is a type of external:InterpreterObject#interpreterMethod that parses another interpreter function, possibly surrounded by padding text that can be parsed by optional leading and trailing regular expressions defined in its factory InterpreterMethodFactory#wrap. If any of these fails to parse, then the whole interpreter method fails to parse and the text pointer is restored to where it was when the method started to parse. Also, no interpretations of its descendant parts are run until the the whole interpreter method is successfully parsed.

If the interpreter method was made with an external:ThisBinding#wrapInterpretation, it will call that with the result of its part, and return the result. The interpretation will be called with this bound to the object of the interpreter method. Otherwise it will return the result of its part, untouched.
